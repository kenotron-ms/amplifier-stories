The Origin

From Blueprint to Build

OpenAI's Symphony

A published specification for autonomous software engineering daemons — a blueprint for an AI-driven factory that picks up issues, codes solutions, and opens pull requests without human intervention in the execution loop.

Amphony

A full reimplementation of that spec using Amplifier as the runtime engine. Not a fork. Not a wrapper. A ground-up build that replaces three core dependencies with Amplifier-native equivalents — built in 7 days by a single engineer.

The original spec assumed Linear, Codex, and in-memory state. Amphony swaps all three — proving the spec itself is runtime-agnostic when the architecture is right.

How It Works

The Autonomous Loop

Poll
→
Dispatch
→
Reconcile
→
Retry
→
Complete

🔍 Poll

Every 30 seconds, scan GitHub Issues for eligible work. Apply the 7-gate eligibility check before advancing any issue.

🚀 Dispatch

Claim the issue by labeling it status:in-progress. Spin up an isolated workspace. Launch an Amplifier session with the rendered WORKFLOW.md prompt.

🔄 Reconcile

After each cycle, compare in-memory worker state against DoltDB truth. Detect stale runs, requeue retries, update labels to match actual state.

♻ Retry

Failed runs enter a backoff queue. DoltDB stores retry counts and last-attempt timestamps. The reconciler requeues eligible retries on the next heartbeat.

✅ Complete

Agent commits work, creates a branch, opens a PR via gh pr create, updates the issue label to status:done, stores the PR URL in DoltDB.

Configuration

WORKFLOW.md — One File to Rule Them All

A single WORKFLOW.md with YAML frontmatter + Liquid prompt template configures the entire factory. Hot-reloaded on every dispatch cycle — change it while the service is running.

YAML Frontmatter

--- parsed by orchestrator on every cycle tracker: type: github repo: owner/repo polling_interval: 30 # seconds agent: runtime: amplifier max_turns: 1 turn_timeout_ms: 1800000 # 30 min state_store: type: doltdb db_path: /var/db/amphony ---

Liquid Prompt Template

# rendered per issue at dispatch time You are an expert software engineer. Resolve issue #{{issue.number}}: {{issue.title}} Acceptance criteria: {{issue.body}}
Dispatch Logic

Seven Gates to Dispatch

All seven conditions must pass before an agent is dispatched. One failure = skip and evaluate the next issue.

1
Required fields present — issue has id, identifier, title, and state; malformed issues are skipped silently
2
Active state — state maps to an active label (e.g. status:todo); terminal states are never re-dispatched
3
Not already running — no live worker thread holds this issue; prevents duplicate agents on the same issue
4
Not already claimed — label is not yet status:in-progress; guards against race conditions across restarts
5
Global concurrency slots available — total running workers < configured max_workers ceiling
6
Per-state concurrency slots available — workers in this state < configured per_state_max limit
7
No non-terminal blockers — all issues in depends_on are in terminal states; respects dependency ordering

⚠ All 7 must pass

A single failed gate causes the issue to be skipped. The orchestrator moves to the next candidate immediately.

Reliability

Self-Healing

No error crashes the service.

🌐
GitHub API fails
Skip dispatch cycle, keep all running workers alive. Retry automatically on the next poll interval.
🗃
DoltDB fails
Fall back to in-memory run tracking. The service continues operating; only durability degrades.
📄
Config parse fails
Keep last-known-good WORKFLOW.md in memory. Log the parse error. Never halt the orchestrator.
🪝
Hook fails
before_run / after_run failures are fatal to that run only. Cleanup hooks are best-effort; won't block others.
💥
Process crash & restart
On startup, reads all in-progress runs from DoltDB, reschedules retries, reconciles GitHub labels.
⏳
Agent timeout
turn_timeout fires after 30 min. Run recorded as failed in DoltDB. Enters retry queue with exponential backoff.

State Store

Git for Your AI's Work

DoltDB stores every state mutation as a versioned commit. The complete history of every issue is auditable, time-travelable, and diff-able — like git log for your AI's decisions.

a3f2b1c dispatch owner/repo#42 run 9e1f2a3d-0c4b
b7c9d2e succeeded owner/repo#42 run 9e1f2a3d-0c4b
c1d4e5f pr: owner/repo#42 → https://github.com/owner/repo/pull/7
d8a3b2c dispatch owner/repo#51 run f7e3b2a1-5d8c
e2f1c9d failed owner/repo#51 run f7e3b2a1-5d8c (timeout)
f4a8d7e retry owner/repo#51 attempt=2 run 1b2c3d4e-9f0a
