A Proposal

We're Saving the Code.
We're Losing the Conversation.

Every commit tells a story.
Right now, we only save the ending.

Proposal March 2026
What We Built

Software has always been
more than code.

For decades, we didn't just write code — we built a record of why we wrote it. These artifacts let future engineers understand a codebase without reading every line.

📋

Pull Requests

Every change explained, reviewed, and approved before merge. The reasoning behind the diff.

📐

Design Documents

Architecture decisions recorded before the first line of code. Trade-offs weighed, alternatives considered.

🔍

Code Reviews

Expert critique that caught bugs, improved design, and transferred knowledge between engineers.

🎯

Issue Trackers

Problems documented, solutions discussed, decisions logged. The problem before the solution.

🏗️

Architecture Decisions

ADRs that captured what we chose, what we rejected, and why — a roadmap of intent.

👤

Maintainers

People who carried the institutional memory of why things are the way they are. The last line of defense.

The Erasure

AI bypasses all of it.
All at once.

  • No pull request.
  • No design document.
  • No code review.
  • No issue trail.
  • No institutional memory.
  • Just code.

Every line of AI-generated code that ships without its conversation attached is a debt we are accumulating against our future ability to understand, maintain, and trust the systems we build.

The Symptom

Something breaks.
There is no trail.

🔇

No One to Ask

The AI that wrote it has no memory of the session. The developer who prompted it may not remember what they asked — or why.

🚫

No Trail to Follow

No PR discussion. No design rationale. No issue thread. The code exists without context — an orphan artifact in a codebase full of them.

⚠️

Wrong Diagnosis

Even asking AI to diagnose the problem can produce completely wrong analysis — misidentifying a bug as a security issue, or inventing causes that don't exist.

The Workaround

Developers are improvising.

Across codebases, engineers are independently creating the same pattern: manual Markdown files to preserve context that AI sessions destroy. It's better than nothing. Barely.

# A real pattern emerging in AI-assisted codebases project/ ├── src/ │ ├── auth.py │ ├── api.py │ └── models.py ├── architecture.md ← manual, written after the fact ├── memory.md ← manual, inevitably stale ├── bug-1.md ← manual, one-off ├── decisions/ │ ├── why-we-chose-postgres.md ← manual │ └── auth-flow-rationale.md ← manual └── README.md

Fragmented. Manual. Unsystematic. Each developer invents their own naming convention. None of it captures the actual AI conversation that produced the code.

The Root Cause

The conversation is divorced from the code.

Most AI coding tools store session data outside the project folder — in the user's home directory, never committed, never shared. The reasoning that produced the code is structurally separated from the code itself.

Where the code lives

  • Inside the project folder
  • Under version control
  • Committed to the repository
  • Shared with every collaborator
  • Preserved in git history forever

Where the conversation lives

  • Outside the project folder — in the user's home directory
  • Outside the project folder entirely
  • Never committed to any repository
  • Never shared with collaborators
  • Lost when sessions end or machines change
The Fix

Store the conversation
in the repository.

Co-locate AI chat sessions with the code they produced.
Same repo. Same version control. Same history.

An observation from months of AI-assisted development

An Emerging Consensus

This is not a niche observation.

💡

Independent Discovery

Thomas Dohmke, former CEO of GitHub, just raised $60M for Entire — a CLI layer on top of git that intercepts AI sessions on every commit: what you wrote, what the agent replied, which files changed. Stored as a checkpoint. Searchable alongside your code history.

🏢

Industry Movement

When the former CEO of the platform that hosts most of the world's code starts a company around conversation preservation, it's a signal — not a coincidence.

🔮

The Question

The problem is visible to anyone watching closely enough. The question isn't whether we'll solve it, but whether we'll solve it before the debt becomes unmanageable.

What Teams Do Today · Workarounds

The tools don’t exist.
So we improvise.

Teams working with AI today face two approaches, neither adequate. Some manually write architecture.md, memory.md, bug-1.md — fragmented notes that capture whatever someone thought to write down. Others use automated hooks that extract summaries from the chat into Markdown files. Both are workarounds. Neither is a solution.

🤖 Future Sessions
Agents read memory via system prompt

Why workarounds fall short

  • Manual files capture only what someone remembered to write
  • Automated summaries lose the reasoning chain — the back-and-forth
  • Neither preserves the full conversation
  • Neither is (A) complete or (B) automatic

What we actually need

  • Full session preserved — not just a summary
  • Automatic — no human action required
  • Co-located with the code in version control
  • Searchable and auditable by future contributors
The Real Risk

Not documentation loss.
Reasoning loss.

AI-generated code can look perfectly reasonable and be completely wrong.

Without the conversation that produced it, you cannot audit the reasoning. You cannot verify the assumptions. You cannot tell if the AI hallucinated the approach. You cannot understand what the developer actually asked for.

The code is the answer. We're throwing away the question.

The Path Forward

Make the conversation
a first-class artifact.

1

Store AI chat sessions in the repository

Next to the code, under version control, committed with the changes they produced.

2

Version them alongside the code

When the code changes, the conversation that drove the change travels with it in the same commit.

3

Make them searchable and auditable

Future engineers should be able to find why code was written, not just what it does.

4

Treat reasoning as sacred as the code itself

We would never ship a binary without source. We should never ship code without its reasoning.

We obsess over what AI can generate.
We're not paying attention to what it destroys.
It's time to pay attention.
